# Nemantix Knowledge Base

## 1. High-Level Architecture & Principles

The **Nemantix Knowledge Base (NKB)** is a modular, format-agnostic engine designed to power advanced AI agents with **Graph RAG (Retrieval-Augmented Generation)** capabilities. 

### 1.1 Design Principles
1. **Structural Intelligence:** Unlike traditional RAG which treats documents as flat lists of chunks, NKB deduces the document's logical hierarchy. It uses a Knowledge Graph to navigate context vertically and horizontally.
2. **Format Agnosticism:** Through a specialized plugin system, the NKB treats PDFs, Markdown files, or Text documents identically. All format-specific logic is isolated within drivers, while the core remains unified.
3. **Logical Partitioning:** A relational registry manages logical "Views." This allows the system to partition different documents into distinct scopes, ensuring that an agent only retrieves information it is authorized to see.

### 1.2 Foundational Concepts

To effectively use the Knowledge Base, it is important to understand its three main pillars: **Locations**, **Indexes**, and **Views**.

1. **Location**:  
Nemantix is designed to be fully agnostic regarding where your documents physically reside. Instead of relying on rigid file paths, the framework uses the `Location` object. 

   A `Location` abstracts the physical or remote source of a document (e.g., a local file, an HTTP URL, an AWS S3 bucket). The internal `LocationRegistry` handles fetching the bytes dynamically, ensuring that the parsing plugins always work seamlessly without needing to know the file's origin.

   *Current support includes local paths (`"path"` or `"file"`), with the architecture ready to plug in remote cloud providers seamlessly.*

2. **Index**:  
Indexing is functional for searching and extracting the most useful portions of a document. When you ingest documents into an Index, Nemantix builds the underlying infrastructure to optimize this search:
   * A dedicated **Vector Collection** in your Vector Store (e.g., Qdrant).
   * A dedicated **Knowledge Graph** file on your local storage.

3. **View**:  
A View defines the perimeter of the knowledge base that an AI Agent has access to. It acts as a logical grouping, determining exactly which documents an agent is allowed to query. This allows you to restrict or grant access to specific sets of information across different Indexes without duplicating the underlying data.


### 1.3 Supported Plugins & Limitations
The system is built to be highly extensible via the `DocumentPluginRegistry`. Currently, the following plugins are implemented and supported:
* **Text Plugin (`text_plugin`)**: Supports plain text and Markdown formats (`.txt`, `.md`).

## 2. Ingestion Guide

In the Nemantix ecosystem, **ingesting** a document means transforming a raw file into structured, searchable intelligence. It is the comprehensive process of reading, processing, and indexing the data across both the Vector Store and the Knowledge Graph, preparing it for AI-driven retrieval.

### 2.1 The Data Journey
1. **Planning:** An LLM analyzes the document layout and creates a structural tree.
2. **Segmentation:** The document is cut into non-overlapping chunks, recovering "orphaned text" between structural boundaries.
3. **Enrichment:** Chunks are summarized (creating the "Text View") and metadata are extracted.
4. **Graph Construction:** The tree is converted into a Directed Graph.
5. **Vectorization:** High-dimensional embeddings are generated and stored with flattened metadata.


The ingestion process is managed by the `KnowledgeBaseManager`. It orchestrates the transition from a raw file to a set of vectors in the database and a graph file.

### 2.2 Configuration Parameters (`KnowledgeBaseManagerConfig`)

The `KnowledgeBaseManagerConfig` object controls every aspect of the ingestion pipeline, from local storage paths to LLM context windows and database connections. 

Here is a breakdown of the available parameters grouped by category:

#### Storage & Paths
These parameters define where local artifacts (like Knowledge Graphs and local Vector DB files) are saved.
* **`base_storage_path`** *(default: "kb_storage")*: The root directory for all physical artifacts generated by the Knowledge Base.
* **`vector_subdir`** *(default: "vector_db")*: Subdirectory for storing local vector collections (if using a local vector store).
* **`graph_subdir`** *(default: "graphs")*: Subdirectory where the serialized NetworkX Knowledge Graphs (`.pkl` files) are saved.

#### Debugging
* **`enable_debug`** *(default: False)*: If set to `True`, the engine saves the intermediate outputs (like the LLM's structural planning and the enriched items) as JSON files. 
* **`debug_subdir`** *(default: "debug")*: Subdirectory for the JSON debug files.

#### Pipeline Parameters 
These parameters tune the behavior of the LLM during the extraction and summarization phases.
* **`planner_window_lines`** *(default: 1000)*: The maximum number of lines passed to the LLM in a single prompt during the hierarchy planning phase. Adjust this based on your LLM's context window.
* **`planner_overlap_lines`** *(default: 200)*: The number of lines that overlap between consecutive windows to ensure no structural context is lost at the boundaries.
* **`enricher_max_workers`** *(default: 20)*: The maximum number of concurrent threads used during the bottom-up summarization phase. Higher values speed up ingestion but may hit LLM API rate limits.

#### Vector Store & Embeddings
* **`embedder_model`** *(default: "sentence-transformers/all-MiniLM-L6-v2")*: The model used to generate vector embeddings for the extracted text chunks.
* **`vector_store_type`** *(default: "qdrant")*: The backend used for the Vector Store (e.g., `qdrant`).

#### Relational Registry (PostgreSQL)
Credentials and connection details for the relational database, which manages Views, document tracking, and idempotency.
* **`db_engine`** *(default: "postgresql")*: The database engine dialect.
* **`db_host`** *(default: "localhost")*: Database host address.
* **`db_port`** *(default: 5432)*: Database port.
* **`db_database`** *(default: "nemantix_db")*: The name of the target database.
* **`db_username`** : Database user.
* **`db_password`** : Database password.

### 2.3 Ingestion Example
The following script demonstrates how to configure and run the ingestion for a single document.

> **Prerequisite:** The Knowledge Base Manager relies on a relational registry to track documents, indexes and logical views. You must have a running **PostgreSQL** instance accessible (matching your DB configuration parameters) before executing the ingestion pipeline.

```python
from nemantix.knowledge_base.document_structure.location import Location
from nemantix.knowledge_base.core.knowledge_base_manager import KnowledgeBaseManager, KnowledgeBaseManagerConfig
from nemantix.core import Expertise

from dotenv import load_dotenv

load_dotenv()

def run_ingestion():
    # Initialize the LLM Proxy
    llm = Expertise.get_default_llm(
        vendor="openai", 
        model="gpt-5-mini", 
    )

    # Configure the Knowledge Base Manager
    config = KnowledgeBaseManagerConfig(
        base_storage_path="kb_storage",
      
        planner_window_lines=1000,   # Lines per LLM analysis window
        planner_overlap_lines=200,    # Overlap to maintain context
        enricher_max_workers=20,      # Parallel enrichment tasks
        
        embedder_model="sentence-transformers/all-MiniLM-L6-v2",
        
        vector_store_type="qdrant",
        vector_subdir="qdrant_db",
      
        db_host="localhost",
        db_password="xxx",
        db_username="xxx",
        db_database="example",
        db_port=5432,
      
        enable_debug=True,
        debug_subdir="debug",
    )

    # Instantiate the Manager
    kb_manager = KnowledgeBaseManager(llm=llm, config=config)


    # index_name: Defined the physical collection in the vector store and the graph name
    # target_views: Logical scopes for retrieval permissions
    kb_manager.index_document(
        location=Location("file", "docs/manual.md"),
        index_name="company_docs",
        target_views=[
            {
                "view_id": "technical_staff",
                "name": "Technical Staff",
                "description": "Internal documentation for the engineering and tech team"
            }
        ],
        doc_type="manual"
    )


if __name__ == "__main__":
    run_ingestion()
```

You can also process all files within a folder in a single call.
```python
kb_manager.process_folder(
            folder_path=target_folder,
            index_name="company_docs",
            target_views=[
                {
                    "view_id": "technical_staff",
                    "name": "Technical Staff",
                    "description": "Internal documentation for the engineering and tech team"
                }
            ],
            doc_type="guide"
        )
```

Keep in mind that specifying a doc_type will apply it to all documents within that folder.
## 3. Agent Integration

Once documents have been ingested and indexed, the Knowledge Base can be seamlessly connected to a Nemantix AI Agent. This allows the Agent to autonomously query the vector database and navigate the structural Knowledge Graph to answer user requests.

### 3.1 Configuration Parameters (`KnowledgeBaseConfig`)

To enable the Knowledge Base, you must instantiate a `KnowledgeBaseConfig` for retrieval and pass it to the `Agent` during initialization. 

The KnowledgeBaseConfig object defines the access perimeter and the connection settings for both the relational registry and the physical storage. It is the primary configuration used to initialize the Knowledge Base environment.

#### Access Control

* **`view_ids`**: A list of strings representing the logical Views the agent or process can access. This defines the "security perimeter" of the knowledge retrieval.

#### Relational Registry

These parameters manage the connection to the database that tracks document metadata, hashes (idempotency), and logical view associations.

* **`db_engine`** (default: "postgresql"): The database engine dialect.
* **`db_host`** (default: "localhost"): The database server address.
* **`db_port`** (default: 5432): The connection port.
* **`db_database`** (default: "nemantix_db"): The name of the target database schema.
* **`db_username`** : The database user.
* **`db_password`**: The password for the specified user.

#### Storage & Vector Store

These settings define where the physical search artifacts are located and which technology powers the semantic retrieval.

* **`base_storage_path`** (default: "kb_storage"): The root directory on the filesystem where all persistent artifacts (graphs, local vector files) are stored.
* **`vector_subdir`** (default: "vector_db"): The specific subdirectory within the base path dedicated to the Vector Store data.
* **`vector_store_type`** (default: "qdrant"): The backend engine used for semantic search (e.g., qdrant).

### 3.2 Knowledge Base Integration Example

**Crucial Note on Scope:** During ingestion, the `KnowledgeBaseManager` uses the `target_views` parameter to group documents. During retrieval, the `KnowledgeBaseConfig` uses the `view_ids` parameter. This acts as a strict authorization filter: the Agent will only be able to search and retrieve information from documents that belong to the specified `view_ids`.

```python
from nemantix.core import Expertise, Agent
from nemantix.knowledge_base.core.nemantix_knowledge_base import KnowledgeBaseConfig
from nemantix.llm.abstract_proxy import AbstractLLMProxy
from nemantix.llm.factory import LLMProxyFactory
from nemantix.llm.credentials import Credentials

creds = Credentials()  # optional; will fallback to env if missing
AbstractLLMProxy.set_credentials_manager(creds)

llm_proxy = LLMProxyFactory.create_llm_proxy(
        "openai",
        model_name="gpt-5-mini",
    )

# Configure the Knowledge Base Retrieval Scope
kb_config = KnowledgeBaseConfig(
    view_ids=["technical_staff"],         # The Agent's authorization scope
    vector_store_type="qdrant",
    vector_subdir="qdrant_db",
    db_host="localhost",
    db_password="xxx",
    db_username="xxx",
    db_database="example",
    db_port=5432
)

exp = Expertise(...)

agent = Agent(
    expertise=exp, 
    llm_proxy=llm_proxy, 
    use_knowledge_base=True,      # Activate the KB module
    kb_config=kb_config           # Pass the retrieval configuration
)
```

It is also possible to delete an index

```python
kb_manager.delete_index("index_name")
```

## 4. NXS Built-in Functions for Graph RAG

To allow AI Agents to interact seamlessly with the Knowledge Base from within their Expertise scripts (.nxs), the system exposes four native built-in functions. These functions abstract the complexity of vector mathematics and graph theory, providing the Agent with an intuitive toolkit to search, zoom in, zoom out, and navigate document contexts.

### retrieve (Semantic Entry Point)

The retrieve function is the primary gateway to the Knowledge Base. Unlike simple vector searches, it operates across a multi-index environment restricted by the Agent's specific Views.

It identifies all Documents mapped to the current Agent's view_ids, and applies a mandatory Agnostic Filter to ensure the agent never retrieves data outside its authorized perimeter. 

Parameters:
* `query`: The search string or question.
* `k` (default=5): The maximum number of results to return.
 
It returns a list of extracted items, each containing:
* `score`: The semantic similarity score
* `breadcrumbs`: The full hierarchical path (e.g., Book > Chapter 1 > Section A).
* `node_id`: The unique ID in the Graph. Required to use expand, generalize, or extend.
* `content`: The processed text 

### expand (Vertical Top-Down Traversal)

The expand function allows for "Zooming In" on a specific portion of the document.  
It navigates vertically downward through the Knowledge Graph: given a node_id, it locates the specific Knowledge Graph file containing that node and retrieves all its direct children.

Parameters:

* `node_id`: The unique identifier of the target node.


### generalize (Vertical Bottom-Up Traversal)

The generalize function is used for "Zooming Out" to gain broader context.  
Given a node_id, it navigates vertically upward to the parent node.

Parameters:

* `node_id`: The unique identifier of the target node.

### extend (Horizontal Traversal)

The extend function enables "Linear Navigation," allowing the agent to move horizontally across the document's flow.  
It identifies the siblings of the current node (nodes sharing the same parent) and identifies the immediate predecessor and successor in the chronological order of the document.

Parameters:

* `node_id`: The unique identifier of the target node.

## 5. Experimental Coding Support
The capabilities of the knowledge base can be used to retrieve 
additional context used by the coding phase. This is an experimental 
feature that can be optionally enabled in the `Expertise`, for example:
```python
from nemantix.core import Expertise

exp = Expertise.from_local_scripts(paths=[...],
                                   verifier=...,
                                   observers=[...],
                                   model='gpt-5-mini', vendor='openai',
                                   # if true, will use the KB assigned to Agent
                                   experimental_enhance_coding=True)
```
If enabled, additional context is retrieved during the coding of a `deliberate` according to its guidelines.
